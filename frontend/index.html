System Flow: End-to-End Request Processing
Complete technical breakdown of the 7-step pipeline • Total latency: 200-800ms
START Client Request
Endpoint: POST /api/v1/chat
Payload: { user_id, message, session_id }
Entry Point: src/api/routes/chat.py → process_chat_simplified()
↓
STEP 1 Store Raw Chat Event
Action: INSERT INTO app.chat_events
Purpose: Immutable audit trail (Layer 1)
Idempotency: SHA256(content) prevents duplicates
Time: ~5ms
↓
PHASE 1A Entity Resolution (5-Stage Hybrid)
Goal: Resolve "Kai" → customer:kai_media_123 • Cost: ~$0.0003 • Time: 30-200ms

5 Stages (fast → slow, stop at first match):
1️⃣ Exact Match: SELECT FROM canonical_entities WHERE canonical_name = 'Kai'
2️⃣ Alias Lookup: SELECT FROM entity_aliases WHERE alias_text = 'Kai'
3️⃣ Fuzzy Match: similarity('Kai', canonical_name) > 0.7 (pg_trgm)
4️⃣ LLM Coreference: "they" → previous entity in conversation (5% of cases)
5️⃣ Domain DB: SELECT FROM domain.customers WHERE name ILIKE '%Kai%'

Output: ResolvedEntity(entity_id, confidence, method, metadata)
↓
PHASE 1B Semantic Extraction (LLM Triple Extraction)
Goal: Extract facts as (subject, predicate, object) triples • Cost: ~$0.001 • Time: 200-500ms
Model: gpt-4o-mini

Example: "Kai prefers delivery on Fridays" →
  • subject_entity_id: customer:kai_media_123
  • predicate: "delivery_preference"
  • object_value: "Friday"
  • confidence: 0.85
  • INSERT INTO app.semantic_memories

Conflict Detection: Compare new facts vs existing semantic_memories + domain.customers
Output: List of SemanticMemory objects + detected conflicts
↓
PHASE 1C Domain Augmentation (SQL Queries)
Goal: Fetch authoritative business data • Cost: $0 • Time: 10-50ms

Query Triggers:
• Customer: When entity → customer:* | SELECT * FROM domain.customers
• Invoice: When "invoice" OR "payment" mentioned | SELECT * FROM domain.invoices WHERE status = 'open'
• Sales Order: When "order" mentioned | SELECT * FROM domain.sales_orders
• Work Order: When "task" OR "work order" mentioned | SELECT * FROM domain.work_orders

Output: List of DomainFact(type, entity_id, content, metadata, source_table)
↓
PHASE 1D Memory Scoring (Multi-Signal Ranking)
Goal: Rank memories by relevance to query • Cost: $0 • Time: 10-30ms

Scoring Formula:
relevance = 0.40 × semantic_similarity (pgvector cosine)
            + 0.25 × entity_overlap (Jaccard similarity)
            + 0.20 × recency (exp(-days × decay_rate))
            + 0.15 × confidence (base × validation_decay)

Output: Top 10 memories ranked by relevance_score (0.0-1.0)
↓
STEP 6 Generate LLM Reply
Input Context:
• User query
• Resolved entities (from Phase 1A)
• Domain facts (from Phase 1C)
• Top memories (from Phase 1D)
• Conversation history (last 10 messages)

Model: gpt-4o-mini • Time: 100-300ms
Output: Natural language reply synthesizing all context
↓
END JSON Response with Full Provenance
Response Structure:
{
  "response": "Natural language reply...",
  "augmentation": { domain_facts, memories_retrieved, entities_resolved },
  "memories_created": [{ memory_type, event_id, content }],
  "provenance": { memory_ids, similarity_scores, source_types },
  "conflicts": [{ subject, existing_value, new_value, resolution }]
}
Memory Storage & Creation: Deep Dive
How the system stores, creates, and manages memories throughout the pipeline

📦 Where Memories Are Stored

Database Schema: 10 Essential Tables

Layer 1: Raw Events
• app.chat_events - Immutable audit trail of all conversations
  • Stores: user_id, session_id, role, content, content_hash, created_at
  • Purpose: Complete provenance (who said what, when)

Layer 2: Entity Resolution
• app.canonical_entities - Resolved entities with canonical names
  • Stores: entity_id (PK), canonical_name, entity_type, created_at
  • Example: "customer:acme_123" → "Acme Corporation"
• app.entity_aliases - Learned aliases for entities
  • Stores: alias_text → canonical_entity_id (FK), user_id, confidence
  • Example: "Acme" → customer:acme_123, "Kai" → customer:kai_media_123

Layer 3: Episodic Memories
• app.episodic_memories - Event-based memories (not yet implemented)
  • Stores: user_id, event_id (FK), summary, entities, created_at
  • Purpose: "User asked about invoice INV-1234 on March 10th"

Layer 4: Semantic Memories
• app.semantic_memories - Fact-based memories (SPO triples)
  • Stores: subject_entity_id (FK), predicate, object_value (JSONB), confidence
  • Example: customer:acme_123 | "delivery_preference" | {"day": "Friday"}
  • Status lifecycle: active → deprecated → conflicted
  • Confidence decay: confidence × e^(-age × decay_rate)

Layer 5: Procedural Memories
• app.procedural_memories - Pattern-based memories (Phase 1D)
  • Stores: pattern_name, condition, action, confidence
  • Example: "When delivery mentioned → check invoices"

Layer 6: Memory Summaries
• app.memory_summaries - Consolidated summaries (Phase 2)
  • Stores: entity_id (FK), summary_text, source_memory_ids, created_at
  • Example: "Acme Corp profile (3 sessions, 12 facts)"

Supporting Tables:
• app.memory_conflicts - Tracks contradicting facts
• app.domain_ontology - Business domain schema
• app.system_config - Tunable heuristics

⚙️ How Memories Are Created (By Pipeline Stage)

Stage 1: Store Chat Event
src/demo/api/chat.py → send_chat_message()
• Creates ChatEvent record immediately on request
• Content hashed (SHA256) for deduplication
• Both user message AND assistant reply stored
• Database: INSERT INTO app.chat_events

Stage 2: Entity Resolution (Phase 1A)
src/domain/services/entity_resolution_service.py
• Creates CanonicalEntity on first mention (lazy creation)
• Creates EntityAlias when fuzzy match succeeds
• Learning: User-specific aliases stored for future resolution
• Database: INSERT INTO app.canonical_entities, app.entity_aliases

Stage 3: Semantic Extraction (Phase 1B)
src/application/use_cases/extract_semantics.py
• LLM extracts SPO triples from message
• Creates SemanticMemory records (batch INSERT)
• Each triple: subject_entity_id + predicate + object_value
• Initial confidence: 0.85 (explicit), 0.70 (implicit), 0.50 (inferred)
• Database: INSERT INTO app.semantic_memories

Stage 4: Conflict Detection
src/domain/services/conflict_detection_service.py
• Compares new facts vs existing semantic_memories
• Compares new facts vs authoritative domain.customers
• Creates MemoryConflict record if contradiction found
• Resolution strategies: trust_db, trust_recent, ask_user
• Database: INSERT INTO app.memory_conflicts

Stage 5: Memory Validation
src/domain/services/memory_validation_service.py
• Reinforces confidence when memory is validated (confidence +0.05, capped at 0.95)
• Marks memory as deprecated if contradicted
• Updates last_validated_at timestamp
• Database: UPDATE app.semantic_memories SET confidence, last_validated_at, status

Future: Procedural Memory Creation (Phase 1D)
• Extracts patterns from repeated user behaviors
• Creates ProceduralMemory when pattern confidence > 0.75
• Example: "User always asks about invoices after mentioning delivery"
• Database: INSERT INTO app.procedural_memories

🔄 Memory Lifecycle Management

State Transitions (Semantic Memories):

ACTIVE → Initial state when fact is extracted
  ↓ (confidence decays over time without validation)
ACTIVE → Confidence reinforced when fact is used (+0.05, max 0.95)
  ↓ (conflict detected with newer information)
CONFLICTED → Marked when contradiction found
  ↓ (resolved in favor of newer/more authoritative source)
DEPRECATED → Fact superseded, kept for audit trail

Confidence Decay Formula (Passive Computation):
effective_confidence = base_confidence × e^(-days_since_validation × decay_rate)
• decay_rate: 0.0115 per day (configurable)
• Never pre-computed - calculated on-demand when scoring memories
• Ensures freshness without background jobs

Retrieval & Scoring:
• Phase 1D: Multi-signal relevance scoring
• Formula: 0.40×semantic + 0.25×entity_overlap + 0.20×recency + 0.15×confidence
• Semantic similarity: pgvector cosine distance on embeddings
• Entity overlap: Jaccard similarity between entity sets
• Recency: Exponential decay based on created_at
• Confidence: Effective confidence after decay

🎯 Key Design Decisions

1. Surgical LLM Integration
• LLM only for semantic extraction (Phase 1B) and coreference (Phase 1A Stage 4)
• Entity resolution: 95% deterministic (pg_trgm fuzzy match), 5% LLM
• Cost: ~$0.002 per turn vs $0.03 for pure-LLM approach

2. Passive Computation (No Background Jobs)
• Confidence decay calculated on-demand, not pre-computed
• Simpler, more accurate, avoids staleness
• Pre-computation is Phase 2 optimization

3. Epistemic Humility
• Max confidence: 0.95 (never 100% certain)
• All conflicts logged to memory_conflicts table
• Resolution strategy explicitly tracked
• System admits uncertainty when confidence < 0.70

4. JSONB for Context-Specific Data
• object_value in semantic_memories: flexible schema
• confidence_factors: evidence varies per memory
• metadata in entity_aliases: disambiguation context
• Allows rich context without schema migrations

5. Dual Truth Architecture
• Domain database (domain.*): Authoritative business facts
• Memory system (app.*): Contextual understanding
• Both are sources of truth - neither alone is sufficient
• Conflicts resolved via explicit strategy, never silently
